Behavioral Design Patterns Document

Bank Management System Java application. Behavioral design pattern assignment with class diagrams.

Executive Summary

In this document, I explain the behavioral design pattern problems I found in the Bank Management System and how I solved each one using the correct pattern. Template Method, Mediator, Strategy, Observer, and Iterator are behavioral patterns. Singleton is creational, but I included it with the OverDraft use case because it works together with Observer there.

For each case, I describe the issue, the pattern solution, the class roles, and the UML diagram that represents the implementation.

Design Approach

I focused on the real problem each part of the system needed to solve. Then I matched each problem with the design pattern that best fits the responsibility of the classes involved.

The criteria I used were:

  • Template Method: a base class must define a stable algorithm skeleton and delegate variable steps to subclasses or hooks.
  • Mediator: a coordinator must centralize interaction among peer components so peers do not directly orchestrate each other.
  • Strategy: interchangeable algorithms must share an interface and be replaceable by client code without rewriting the caller.
  • Observer + Singleton: there must be listener registration/notification, and the event source must enforce a single shared instance.
  • Iterator: traversal must be exposed through an iterator abstraction that hides the aggregate structure, not just a plain loop.

Each case includes the issue I found, the pattern I used to solve it, the role of each class, and the UML diagram.

Findings

1. Template Method: AbstractTransactionTemplate

Pattern type: behavioral.

Issue I found: deposits, withdrawals, and transfers share a fixed transaction lifecycle: validate input, validate account state, check transaction-specific rules, then apply financial impact. Each transaction type varies only the hook implementations.

Pattern solution: I solved this using Template Method. AbstractTransactionTemplate defines the transaction workflow skeleton. In the current source, this implementation appears as AbstractTransactionProcessor.executeTransaction(...), a final template method that fixes the workflow order and delegates variable steps to abstract methods.

7. Observer + Singleton: OverDraft Use Case

Pattern type: Observer is behavioral; Singleton is creational.

Issue I found: an overdraft event should trigger multiple independent reactions: database recording, customer notification logging, audit logging, and active admin UI refresh. The overdraft detector should not know every reaction.

Pattern solution: I solved this using Observer with Singleton. OverdraftNotificationService maintains a listener list, supports subscription and unsubscription, and notifies every listener on publish. It is also an eager Singleton with a private constructor and static instance accessor.

8. Iterator: Transaction History Filtering

Pattern type: behavioral.

Issue I found: transaction history needs filtered sequential traversal over a list of transactions while hiding traversal and filtering logic from the controller.

Pattern solution: I solved this using Iterator. TransactionCollection creates a TransactionIterator. AccountTransactionHistory hides the internal list. FilteredTransactionIterator tracks traversal state and exposes hasNext/next. The controller consumes the iterator abstraction.

Observer + Singleton Combination Section

How Observer is used

I used OverdraftNotificationService as the Subject/Event Hub. It stores observers in List<OverdraftAlertListener>, exposes subscribe and unsubscribe, and calls observer.onOverdraft(event) for each registered listener. The listeners implement a common callback interface and perform separate side effects.

How Singleton is used

The same publisher is an eager Singleton: OverdraftNotificationService.java:9 declares a private static final instance, OverdraftNotificationService.java:13-17 has a private constructor, and OverdraftNotificationService.java:19-21 exposes the shared instance through getInstance(). Clients publish and subscribe through OverdraftNotificationService.getInstance() in OverdraftMonitor.java:44 and OverdraftAlertView.java:43/86.

Conclusion

These are the final design pattern cases I used in the assignment:

  • Template Method: 2 cases.
  • Mediator: 2 cases.
  • Strategy: 2 cases.
  • Observer + Singleton: 1 combined case.
  • Iterator: 1 case.

Overall, these patterns helped separate responsibilities in the system. Template Method keeps repeated workflows consistent, Mediator centralizes complicated coordination, Strategy separates interchangeable calculations and decisions, Observer with Singleton handles overdraft notifications, and Iterator hides transaction-history traversal details.
